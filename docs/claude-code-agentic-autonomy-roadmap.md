# Claude Code 后端：自主 Agentic 能力开发路线

**文档性质**：开发规划与实施顺序说明  
**范围**：基于当前 `claude-code`（`AgentCore`、工具系统、Sidecar、Cron、权限、Skills 等）向「自主思考、自主完成任务」演进时建议做的工作与落地顺序。  
**最后更新**：2026-04-17（增补议程层、横切观测与评测）

---

## 1. 背景与目标

当前主路径以 **ReAct 式循环**（模型推理 → 工具调用 → 结果回注）为主，配合 Sidecar、Cron、多 Agent、Skills、Checkpoint 等能力，已具备多步执行基础。

「更自主」通常还需要：**可管理的任务生命周期**、**显式规划与失败后的重规划**、**可控的权限与审计**、以及（按需）**记忆 / MCP / 子 Agent** 等扩展。

---

## 2. 名词：任务模型（Task Model）

**任务模型**指：把「要完成的一件事」从「单次对话或单次 `execute()`」中抽离为系统内 **一等公民**，用结构化数据描述并贯穿其整个生命周期。

典型字段包括（可按产品裁剪）：

| 类别 | 示例字段 |
|------|-----------|
| 身份 | 任务 ID、标题 |
| 目标 | 要达成的结果描述（可结构化） |
| 状态 | 待执行 / 执行中 / 已暂停 / 成功 / 失败 / 已取消 |
| 上下文 | 关联 `sessionId`、工作目录、agent、计划版本 |
| 输入输出 | 约束条件、产物路径、验收结果 |
| 运行数据 | 起止时间、当前步骤、错误信息、重试次数 |

**没有任务模型**时：多次 `execute`、Cron 触发与「同一件长线事」难以统一排队、恢复与观测；**有任务模型**后，才便于做队列、取消、超时、审计以及与 Cron 的统一调度。

---

## 3. 实施前建议对齐的前提（验收边界）

在按下列顺序开发前，建议团队先约定：

- **自主范围**：仅工作区与 MCP，还是包含网络、外部账号等。
- **完成判定**：测试通过、lint、结构化输出、或人工确认等。
- **人机边界（HITL）**：哪些操作必须人工在环。
- **资源上限**：步数、token、费用、墙钟时间等（可与现有环境变量如预算上限结合）。

---

## 4. 议程层：自主「确定要做哪些事」

前文阶段 1～6 中，**阶段 1、3、4、5 的大半**侧重 **治理与执行**（任务如何登记、排队、执行、停止、审计、扩工具）；**阶段 2** 侧重 **题目已定** 之后的拆解与重规划。本节补充 **「题目从哪来、谁决定升格为任务」**，即 **议程层（Agenda）**。

### 4.1 与执行层的分工

| 层次 | 回答的问题 |
|------|-------------|
| **议程层** | 世界发生了什么？要不要做成任务？谁来做？何时做？（**选题**） |
| **Planner + 执行** | 给定任务描述后，如何拆解、调用工具、失败如何改计划？（**做题**） |
| **任务模型 + 队列** | 任务如何持久化、排队、取消、超时？（**治理**） |

三者衔接：**议程层**产出「待执行任务」→ 写入 **任务模型** 进入 **队列** → **Planner** 驱动 **AgentCore** 执行。

### 4.2 「自主选题」在工程上可如何理解

可落地的「自主确定要做的事」**很少依赖无约束的纯模型空想**，通常由两类输入共同决定：

1. **事件接入（信号）**  
   例如：定时（Cron）、Webhook、CI 结果、文件变更、Issue 标签、邮件/MCP 等。它们提供 **「世界发生了什么」**，往往表现为候选 backlog 条目或触发器载荷。

2. **前置职能 / 策略（约束与解读）**  
   在信号进入前后，用 **规则 + 配置**（以及可选的小段模型推理）决定：该信号是否与本系统有关、是否去重、优先级、是否自动入队或仅生成建议。  
   更精确表述：**事件提供信号，职能与策略决定「是否变成任务、变成谁的、何时做」。**

因此可以概括：**自主选题 ≈ 事件接入（等信号来源）+ 前置职能与策略（过滤、排序、升格规则）**；其中「前置职能」宜 **显式、可配置、可审计**，而不是隐含在单次 prompt 里难以回放。

### 4.3 与前端配置的 Agent 绑定（按职责分工）

**可以与前端（或配置层）定义的 Agent 绑定**，且这是推荐产品形态：**每个 Agent 有明确的职责边界（mandate）**，用于收窄「对哪些信号敏感、如何解读、是否自动执行」。

常见绑定维度（可按版本渐进实现）：

| 绑定维度 | 作用 |
|----------|------|
| **职责描述 / Soul（系统提示）** | 在「归纳事件 → 是否建任务 → 任务描述怎么写」时按角色推理，保持行为一致。 |
| **允许关注的信号** | 例如仅某仓库、某路径、某标签的 Issue；其他 Agent 忽略无关事件。 |
| **工具与环境范围** | `cwd`、MCP、工具白名单；避免角色越权（如「文档 Agent」不自动改部署配置）。 |
| **自动启动策略** | 有的 Agent 仅产出「建议任务」；有的在白名单事件 + 白名单工具下可自动入队执行。 |

**概念要点**：自主「想做什么」在选题层应主要通过 **该 Agent 的职责 + 允许的输入源 + 策略** 来收窄；职责是 **必要约束**，但不是 **充分安全条件**。

### 4.4 议程层仍需配套的工程能力

职责与模型无法单独保证「想对了、该做」，建议与议程层同步考虑：

- **规则与过滤器**：在进入 backlog 前过滤明显无关或高危信号。  
- **去重与冷却**：同一 `source + external_id` 或等价键避免重复建任务；短时间重复事件防抖。  
- **默认偏「建议」、关键动作 HITL**：尤其是自动执行写库、推送、对外的操作。  
- **可追溯**：记录「因何事件、经哪条规则/哪一 Agent 策略升格为本任务」，便于审计与评测（误报率、重复率等）。

### 4.5 议程层相关开发工作（与阶段 1～6 的关系）

在阶段 1 的任务模型与队列就绪后，可并行或紧随实现：

- **结构化 backlog / 待办 schema**（来源、优先级、截止时间、关联外部 id、`agent_id` 等）。  
- **事件接入与标准化**：Webhook、定时、（按需）Git/CI、文件监听等，统一为内部事件类型。  
- **与 Agent 配置同步**：前端或配置存储中的 `agent_id`、Soul、订阅范围、自动策略写入侧车或服务端，供议程层读取。  
- **排序与入队**：评分规则、与队列并发策略一致；幂等建任务。  
- **可选：轻量「选题」推理**：对候选列表做结构化输出（urgent / ignore / defer / needs_human），**输出 schema 固定、可记录**，而非自由长文。

## 5. 建议实施顺序（务实路线）

下列顺序强调 **先可控、再聪明**：基础设施不到位时直接堆「规划 / 记忆」容易导致行为不可观测、难以调试、安全风险高。

**「谁决定做哪件事」**见上文 **§4 议程层**；下列阶段在任务已创建或已触发的前提下，侧重执行与治理。

**横切能力**（观测与调试、评测与回归）见 **§6、§7**，与阶段 1～6 **并行推进**：从阶段 1 起就应约定任务 ID 与事件 schema，评测场景库可随阶段 2、议程层逐步扩充，不宜全部堆到项目末期。

### 阶段 1：任务模型 + 队列 + 取消 / 超时

**目标**：任何「自主跑起来」的执行都有明确归属、可排队、可打断、可超时。

**要点**：

- 引入任务实体（见上文「任务模型」）及状态机。
- 多路触发（用户聊天、Cron、未来其他入口）共用 **统一队列或并发策略**（例如交互优先、后台限并发）。
- **取消**：用户或系统中止时，清理进行中的流式请求与权限等待。
- **超时**：单任务墙钟上限、单工具调用超时（与工具元数据配合）。

**说明**：没有本阶段，后续 Planner 与自动批准容易在并发与长任务场景下「乱」。

---

### 阶段 2：Planner + 执行循环 + 失败重规划

**目标**：最小可用的「自主思考」——不只是一步一想，而是 **先规划、再执行、失败则改计划**。

**要点**：

- **显式规划**：复杂任务先产出结构化计划（子任务、依赖、验收标准）。
- **执行循环**：按计划驱动多轮工具与模型交互。
- **重规划**：工具失败、环境变化或验收不通过时，修订计划而非硬跑到底。
- **可选**：轻量「反思 / 自检」步骤（对照验收标准）。

实现上可在 `AgentCore` 外层增加 **Orchestrator（编排层）**，避免把全部逻辑塞进单次 `execute()`。

---

### 阶段 3：细粒度权限与审计

**目标**：在扩大自动执行范围前，把风险关进「可配置、可记录」的笼子里。

**要点**：

- **分级策略**：按工具类型、路径前缀、命令类别等配置允许自动执行的范围（优于全局 `auto-approve`）。
- **敏感操作**：如推送远端、破坏性删除、对外发信等，可强制 HITL。
- **审计日志**：任务 ID、时间、触发源、工具调用摘要与结果，便于排错与追责。

---

### 阶段 4：持久化与 Cron / 长任务统一

**目标**：进程重启、应用重开仍能接续；定时触发与「长任务」共用同一套运行语义。

**要点**：

- 任务状态与关键检查点持久化（路径与格式需与现有 `~/.claude` 约定及 Sidecar 生命周期协调）。
- **Cron** 抽象为「在某一时刻向任务队列提交一次任务」或「触发同一 `TaskRunner`」，避免与聊天路径长期分裂两套逻辑。
- 与现有 Session / Checkpoint 能力对齐：何时快照、何时回滚由任务策略定义。

---

### 阶段 5：MCP 动态管理与更多连接器

**目标**：运行时可扩展「能做什么」，而不只依赖启动时静态配置。

**要点**：

- 补齐或加强 Sidecar 与 MCP 相关的动态连接 / 断开 / 重载能力（以当前代码为准迭代）。
- 工具与 MCP 能力元数据：超时、重试、幂等、失败时的结构化错误供 Planner 使用。

---

### 阶段 6：记忆 / RAG / 子 Agent（按产品优先级渐进）

**目标**：长周期、大上下文或专业化分工场景下的增强，而非第一阶段必选项。

**要点**：

- **记忆**：写入时机、检索策略、过期与容量控制；与现有 Agent memory、Skills 分工清晰。
- **RAG**：大仓库或长文档按需检索，控制上下文体积。
- **子 Agent**：多角色分工、上下文隔离、结果汇总与权限继承策略。

---

## 6. 横切能力：观测与调试

自主系统若无 **可观测性**，排障只能靠猜，等同于黑盒。本项与 **阶段 1（任务模型）**、**阶段 2（Planner）** 强相关：先有稳定 **任务 ID** 与事件通道，再挂载 trace。

### 6.1 结构化事件与任务级 trace

- 在现有流式事件（如 `text` / `tool_use` / `tool_result` / `complete` / `error` 等）之上，扩展或关联 **任务维度** 元数据，例如：
  - 任务 ID、`agent_id`、触发来源（用户 / Cron / Webhook / 议程层规则 id 等）；
  - **计划版本**、子任务 id（若采用 Planner 产出结构化计划）；
  - 每步 **耗时**、累计 **token**（或与现有成本统计对齐）、工具名与摘要（避免泄露敏感参数时可脱敏）。
- 目标：任一任务可从「创建 → 规划 → 每步执行 → 结束」还原时间线与决策依据（在审计策略允许范围内）。

### 6.2 前端与运维体验

- **任务列表**：状态、排队位置、当前步骤、开始/结束时间、关联会话。
- **实时日志 / trace 视图**：可折叠子任务、过滤仅错误、导出（按需）。
- **失败原因与动作**：明确错误分类；**重试**（整任务或从某计划版本）、**取消** 与阶段 1 的取消语义一致。
- **与 Tauri / Sidecar 事件统一**：例如已有 `$/cron` 等通知，新建事件应 **统一命名规范与 payload 结构**（版本字段、任务 id、错误码），避免前端多套解析逻辑。

### 6.3 实施节奏建议

- **与阶段 1 同步**：确定任务实体时即定义 **trace 挂载点** 与 **最小事件 schema**。  
- **随阶段 2 增强**：Planner 引入后补齐计划版本、子任务 id 等字段。  
- **议程层上线后**：trace 中增加「升格为任务的原因」（事件 id、规则 id），便于诊断误触发。

## 7. 横切能力：评测与回归

自主循环与议程层逻辑迭代频繁，若无 **场景库与指标**，极易「改一处、坏一片」。

### 7.1 场景库

- 覆盖典型 **多步任务**：修 bug、加小功能、跑测试与 lint、工具失败后的重试与重规划等。  
- 若已实现 **议程层**：增加「事件 → 是否应建任务」「去重」「误报/漏报」等用例。  
- 场景应断言 **行为与约束**（是否完成、是否调用允许范围内的工具、是否在预算内），而非硬编码某次模型自然语言输出的具体措辞或「标准答案」（违背通用评测原则）。

### 7.2 回归指标（可按版本逐步齐全）

示例维度：

- **执行层**：任务完成率、平均步数、平均墙钟时间、费用与 token。  
- **安全与质量**：误操作率（如越权路径、被策略拦截次数）、权限弹窗次数（交互模式下）。  
- **议程层（若有）**：误建任务率、重复任务率、用户取消率。

### 7.3 与阶段路线的关系

评测不是「阶段 6 之后才做」的收尾项；建议 **早期建立小规模场景库**，随阶段 2、阶段 4、议程层能力上线 **同步扩充** 用例与指标门禁。
